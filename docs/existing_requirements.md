# Uptime Kuma 现有系统需求文档

> 本文档基于运行实例的页面采集（见 `docs/existing_interaction_flows.md`）与项目源代码（`server/`、`src/`）逆向整理，描述系统当前已实现的能力，作为后续演进的现状基线。
>
> 功能需求采用 **EARS**（Easy Approach to Requirements Syntax）格式：
> - **普遍型**：系统应 `<行为>`。
> - **事件驱动**：当 `<触发>` 时，系统应 `<行为>`。
> - **状态驱动**：在 `<状态>` 期间，系统应 `<行为>`。
> - **异常型**：如果 `<条件>`，则系统应 `<行为>`。
> - **可选特性**：在启用 `<特性>` 时，系统应 `<行为>`。

---

## 1. 用户角色定义

Uptime Kuma 采用「单管理员 + 公开访客」的轻量权限模型，不提供细粒度的多用户 RBAC。

| 角色 | 标识 | 描述 | 认证方式 |
| --- | --- | --- | --- |
| 管理员用户 | `Admin` | 系统首次启动时创建的唯一账户，拥有监控项、通知、状态页、维护、设置等全部管理权限 | 用户名/密码（可选 2FA） |
| 公开访客 | `Visitor` | 无需登录，访问已发布的公开状态页（`/status/<slug>`） | 无（公开页）/ 可选页面密码 |
| API 客户端 | `ApiClient` | 通过 API Key 调用指标 / 接口的外部系统 | API Key（HTTP Basic） |
| 推送监控客户端 | `PushClient` | 被监控侧主动上报心跳的程序（Push 类型监控） | Push Token（URL 内） |

> 说明：系统核心为单用户实例；`user` 表通常仅含一个管理员账户，所有「管理员用户」共享同一身份。

## 2. 功能需求（EARS）

### 2.1 认证与账户

- **FR-AUTH-1**：在认证未被禁用时，系统应要求用户在访问仪表盘及管理接口前完成用户名/密码登录。
- **FR-AUTH-2**：当用户提交有效的用户名与密码时，系统应建立已认证会话并放行受保护资源。
- **FR-AUTH-3**：如果用户提交的凭据无效，则系统应拒绝登录并记录失败尝试。
- **FR-AUTH-4**：系统应使用 bcrypt 存储用户口令；当检测到旧式哈希时，系统应在成功登录时自动升级为 bcrypt。
- **FR-AUTH-5**：在启用两步验证（2FA/TOTP）时，系统应在密码校验通过后额外要求有效的一次性验证码。
- **FR-AUTH-6**：系统应允许管理员修改用户名、口令以及启用/停用 2FA。
- **FR-AUTH-7**：在管理员显式禁用认证（`disableAuth`）时，系统应跳过登录直接放行管理界面。
- **FR-AUTH-8**：当管理员从用户菜单选择「退出」时，系统应结束当前会话。

### 2.2 监控项管理

- **FR-MON-1**：系统应允许管理员创建、查看、编辑、暂停/恢复与删除监控项。
- **FR-MON-2**：系统应提供约 30 种监控类型供选择，包括 HTTP(s)、HTTP(s)-关键字、HTTP(s)-JSON 查询、HTTP(s)-Browser Engine、TCP Port、Ping、DNS、Push、手动、分组、gRPC(s)-关键字、Docker 容器、Steam/GameDig 游戏服务器、MQTT、RabbitMQ、Kafka Producer、SNMP、SMTP、SIP Options、Tailscale Ping、Websocket Upgrade、Globalping，以及 MySQL/MariaDB、PostgreSQL、Microsoft SQL Server、Oracle、MongoDB、Redis、Radius 等数据库/服务类型。
- **FR-MON-3**：当管理员创建监控项时，系统应允许配置显示名称、目标地址、心跳间隔、重试次数、请求超时与通知重发间隔。
- **FR-MON-4**：在监控类型为 HTTP(s) 时，系统应允许配置请求方法、请求体编码、请求体、请求头与有效状态码范围（默认 200-299）。
- **FR-MON-5**：在监控类型为 HTTP(s) 时，系统应允许选择身份验证方式（无、HTTP 基础、Bearer Token、OAuth2 客户端凭据、NTLM、mTLS）。
- **FR-MON-6**：系统应允许为监控项配置高级选项，包括忽略 TLS/SSL 错误、反转模式、缓存绕过参数、最大重定向次数、IP 协议（自动/IPv4/IPv6）及响应保存策略。
- **FR-MON-7**：系统应允许将监控项归入分组、附加标签，并填写支持 Markdown 的描述。
- **FR-MON-8**：当管理员为监控项分配通知渠道或代理时，系统应在该监控项状态变化时使用相应配置。

### 2.3 监控执行与心跳

- **FR-EXE-1**：在监控项处于启用状态期间，系统应按配置的心跳间隔周期性检测其目标。
- **FR-EXE-2**：当一次检测成功时，系统应记录一条 `UP` 心跳并附带响应信息（如响应时间）。
- **FR-EXE-3**：如果检测失败且未超过重试次数，则系统应进入 `PENDING` 重试状态而不立即判定故障。
- **FR-EXE-4**：如果检测失败且已达到配置的重试次数，则系统应将监控项标记为 `DOWN` 并触发已分配的通知。
- **FR-EXE-5**：在启用反转模式时，系统应在目标可访问时判定为故障、不可访问时判定为正常。
- **FR-EXE-6**：在监控类型为 Push 时，系统应在约定间隔内未收到推送心跳时判定为故障。
- **FR-EXE-7**：当 HTTPS 目标的 TLS 证书或域名剩余有效期低于配置阈值时，系统应触发到期通知。

### 2.4 通知

- **FR-NOTIF-1**：系统应支持配置多种通知渠道（当前内置约 95 种提供方，涵盖 Email/SMTP、Telegram、Discord、Slack、Microsoft Teams、PagerDuty、Webhook、Apprise、钉钉、飞书、企业微信、Server酱 等）。
- **FR-NOTIF-2**：当管理员保存通知渠道前点击「测试」时，系统应向该渠道发送一条测试消息并反馈结果。
- **FR-NOTIF-3**：当监控项发生 `UP`/`DOWN` 状态变化时，系统应向该监控项已分配且启用的渠道发送通知。
- **FR-NOTIF-4**：在渠道设置「默认开启」时，系统应将其默认应用于新建监控项；在勾选「应用到所有现有监控项」时，系统应追加到全部现有监控项。
- **FR-NOTIF-5**：系统应允许配置 TLS 证书与域名到期的提醒天数阈值（默认 7/14/21 天，可增删）。
- **FR-NOTIF-6**：系统应允许配置弹窗通知的自动关闭时长（成功/失败可分别设置，-1 为不自动关闭，0 为禁用弹窗）。
- **FR-NOTIF-7**：在渠道启用自定义消息模板时，系统应按模板渲染通知内容。

### 2.5 状态页

- **FR-SP-1**：系统应允许管理员创建、编辑与删除状态页，并为每个状态页设置名称与唯一路径 slug（字符限 `a-z 0-9 -`）。
- **FR-SP-2**：系统应允许将监控项分组编排到状态页，并自定义标题、描述、主题与品牌元素。
- **FR-SP-3**：当访客访问 `/status/<slug>` 时，系统应公开展示该状态页的服务状态与历史，无需登录。
- **FR-SP-4**：在状态页设置了访问密码时，系统应要求访客输入正确密码后方可查看。
- **FR-SP-5**：系统应支持将状态页映射到指定域名，并提供 `default` 特殊路径作为默认页。
- **FR-SP-6**：系统应在状态页提供事件/公告与维护信息展示。

### 2.6 维护计划

- **FR-MNT-1**：系统应允许管理员创建维护计划，包含标题、描述（支持 Markdown）、受影响的监控项与展示的状态页。
- **FR-MNT-2**：系统应支持 6 种调度策略：手动启用/禁用、单一维护时间窗口、Cron 表达式、重复-时间间隔、重复-每周计划、重复-每月计划。
- **FR-MNT-3**：当维护时间窗口生效时，系统应将受影响监控项标记为「维护」状态并抑制其故障通知。
- **FR-MNT-4**：系统应允许为维护窗口配置时区、起止时间，并提供时长快捷设置（15 分钟至 24 小时）。
- **FR-MNT-5**：在维护被配置展示于状态页时，系统应在对应状态页公示该维护信息。

### 2.7 系统设置

- **FR-SET-1**：系统应提供集中的设置界面，包含常规、外观、通知、反向代理、标签、监控历史、Docker 宿主、远程浏览器、安全性、API 密钥、代理、关于等子页。
- **FR-SET-2**：系统应允许配置显示时区、服务器时区、入口页面、站点主 URL 及搜索引擎索引可见性。
- **FR-SET-3**：系统应允许配置 Steam API 密钥、Globalping API Token 及 Chrome/Chromium 可执行文件路径（供浏览器引擎类监控使用）。
- **FR-SET-4**：系统应允许管理员配置数据保留（监控历史）策略并清理历史数据。
- **FR-SET-5**：系统应允许管理员管理标签、代理服务器、Docker 宿主与远程浏览器配置。
- **FR-SET-6**：系统应支持备份/导入导出配置数据。
- **FR-SET-7**：系统应支持外观设置，包括主题（浅色/深色）与界面语言切换。

### 2.8 API 与集成

- **FR-API-1**：系统应允许管理员创建带名称、有效期与启用状态的 API Key，并在创建时一次性展示完整密钥。
- **FR-API-2**：当请求携带有效且未过期的 API Key 时，系统应放行受 API Key 保护的接口。
- **FR-API-3**：如果 API Key 无效、已停用或已过期，则系统应拒绝该 API 请求。
- **FR-API-4**：在启用 API Key 功能时，系统应使用 API Key 校验受保护 API；未启用时回退到基础认证。
- **FR-API-5**：系统应以 Prometheus 兼容格式暴露监控指标供采集。
- **FR-API-6**：系统应为状态页提供供外部消费的徽章（badge）与数据接口。

### 2.9 仪表盘与可视化

- **FR-DASH-1**：当管理员登录后，系统应展示包含「状态速览」（正常/故障/维护/未知/暂停计数）与重要事件列表的仪表盘。
- **FR-DASH-2**：系统应提供监控项搜索框，以及按状态、标签筛选监控项的能力。
- **FR-DASH-3**：当管理员打开某监控项详情时，系统应展示其响应时间图表、可用率与心跳历史。
- **FR-DASH-4**：当管理员点击「清除所有事件」时，系统应清空重要事件列表。
- **FR-DASH-5**：在数据通过 WebSocket 实时推送期间，系统应在状态变化时即时更新界面而无需刷新。

## 3. 非功能需求

### 3.1 性能与可扩展性

- **NFR-PERF-1**：系统应支持最低 20 秒的监控心跳间隔。
- **NFR-PERF-2**：系统应基于 WebSocket（Socket.IO）进行前后端实时通信，状态变化应近实时反映到界面。
- **NFR-PERF-3**：系统应对历史心跳数据进行聚合（按小时/天统计），以支撑长周期可用率与图表查询。
- **NFR-PERF-4**：系统应通过 HTTP 响应压缩（gzip）降低前端资源传输体积。
- **NFR-PERF-5**：系统应支持单实例管理数百个监控项；监控检测应并发执行，互不阻塞。

### 3.2 安全性

- **NFR-SEC-1**：系统应使用 bcrypt 对用户口令进行加盐哈希存储，不以明文保存。
- **NFR-SEC-2**：系统应对登录、API 与 2FA 请求实施速率限制（分别约 20/分钟、60/分钟、30/分钟），超限时拒绝请求。
- **NFR-SEC-3**：系统应支持基于 TOTP 的两步验证。
- **NFR-SEC-4**：系统应对敏感凭据输入（令牌、密钥、密码）采用隐藏输入控件并支持可见性切换。
- **NFR-SEC-5**：API Key 应支持有效期与停用，过期或停用后立即失效。
- **NFR-SEC-6**：系统应在前端对用户提供的 Markdown/HTML 内容进行净化（sanitize）以防止 XSS。
- **NFR-SEC-7**：系统应允许通过反向代理部署，并支持配置站点主 URL 与受信代理。
- **NFR-SEC-8**：在用户未明确公开时，系统默认应阻止搜索引擎索引（noindex）。

### 3.3 可用性与可靠性

- **NFR-AVL-1**：系统应支持优雅关闭（graceful shutdown），在进程退出前完成进行中的请求处理。
- **NFR-AVL-2**：系统应使用持久化数据库（默认 SQLite，亦支持内嵌/外部 MariaDB/MySQL）保存配置与历史，重启后数据不丢失。
- **NFR-AVL-3**：系统应在启动时自动执行数据库迁移以保证 schema 与版本一致。
- **NFR-AVL-4**：单个监控项检测失败不应影响其他监控项或整体服务运行。
- **NFR-AVL-5**：系统应提供容器化部署（Docker），并以单容器、最小配置即可运行。

### 3.4 易用性与可访问性

- **NFR-USE-1**：系统应提供多语言界面（当前约 77 种语言）。
- **NFR-USE-2**：系统应提供浅色/深色主题与响应式布局。
- **NFR-USE-3**：界面应通过表单内的帮助文本、占位符与默认值降低配置门槛。
- **NFR-USE-4**：关键配置应可在前端完成，尽量避免依赖环境变量（仅启动相关项除外）。

### 3.5 可维护性与可移植性

- **NFR-MAINT-1**：系统应运行于 Node.js ≥ 20.4.0。
- **NFR-MAINT-2**：系统应支持 x86_64 / arm64 / armv7 架构，且非 Docker 安装无需原生构建依赖。
- **NFR-MAINT-3**：监控类型与通知渠道应以模块化方式组织（`server/monitor-types/`、`server/notification-providers/`），便于扩展。

## 4. 约束与假设

- 系统为单管理员模型，不含多租户或细粒度角色权限。
- 不支持 NFS 等网络文件系统作为数据目录。
- 公开状态页面向匿名访客开放，其可见性由管理员控制。
- 本文档描述「现状」，统计数值（95 通知渠道、约 30 监控类型、77 语言、13 设置子页、6 维护策略）以采集时的源代码与运行实例为准，后续版本可能变化。
